# Settle — Correctness Criteria

## Input Manifest

### Available at Settle entry

| Artifact | Path | Source |
|----------|------|--------|
| Gherkin spec (primary + subspecs) | `tests/features/{feature_id}/` | Written by Inscribe/Layout |
| Green-phase implementation | `src/` | Written by Realize |
| All tests (passing) | `tests/unit/`, `tests/integration/` | Written by Etch |
| Etch map | `.haileris/features/{feature_id}/etch-map.yaml` | Written by Etch |
| Realize map | `.haileris/features/{feature_id}/realize-map.yaml` | Written by Realize |
| Constitution (when present) | `.haileris/project/constitution.md` | Project-wide |
| Pipeline config (when present) | `.haileris/project/config.{ext}` | Project-wide; governs `settle_loops` |
| Verify report (most recent) | `.haileris/features/{feature_id}/verify_{timestamp}.md` | Written by Inspect |
| Delivery order | `.haileris/features/{feature_id}/delivery-order.yaml` | Written by Layout (regenerated by Settle.Scope) |
| Pipeline state | `.haileris/features/{feature_id}/pipeline-state.yaml` | Shows `inspect: passed` |
| Ascertainments | `.haileris/features/{feature_id}/ascertainments.md` | Written by Ascertain |

Settle is the final stage — every upstream artifact is present. Settle modifies existing artifacts based on findings.

### Artifacts modified by this stage

| Artifact | Condition |
|----------|-----------|
| Source code (`src/`) | `domain: impl` findings |
| Test files (`tests/`) | `domain: test` findings, via three-tier policy |
| Gherkin spec (`tests/features/`) | `domain: spec` findings, via auto-resolve |
| Etch map | When genuinely wrong test fixes change mappings |
| Ascertainments | When `auto_resolve_spec` enabled: `domain: spec` resolutions appended with `[AUTO-RESOLVED]` tag |
| Delivery order | Regenerated by Settle.Scope from current declarations |
| Pipeline state | `rerun_scope`, `loop_count`, subspec statuses updated |

### Read scope

Settle reads the most recent `verify_{timestamp}.md` (sole source of findings), etch-map and realize-map (for BID → subspec ownership), delivery order and subspec `Requires:`/`Provides:` (for dependency graph and blast radius), and pipeline state (for `loop_count`, `subspec_statuses`, `provides_hash`).

## Behavioral Constraints

### Finding scope

Settle processes Critical, High, and Medium findings only. Low and Nit are informational — Settle passes over them.

### Fix authority by resolution domain

| Domain | What Settle changes | Authority level |
|--------|---------------------|-----------------|
| `impl` | Production code in `src/` | Direct fix — targeted to listed findings only |
| `test` | Test files in `tests/` | Three-tier policy (see below) |
| `spec` | Gherkin spec files | When `auto_resolve_spec` is enabled: auto-resolve with `[AUTO-RESOLVED]` tag; append to `ascertainments.md`. When disabled (default): escalate to user. |

### Three-tier test-domain fix policy

1. **Structural refactors** — changes to how a test is written that preserve what it verifies (fixture dedup, setup reorganization, assertion style). Apply directly. Test function names and count are preserved.

2. **Assertion-level corrections** — adding missing assertions or correcting expected values. Closed derivation scope: use only (a) the BID's Gherkin Then/And step and (b) the test's own Arrange section. Apply the fix **and notify the user** with: BID reference, change made, Gherkin step, Arrange values used. Route to Etch for regeneration when derivation requires knowledge beyond these two sources.

3. **Genuinely wrong tests** — wrong interface, unrealistic fixture, mismatched assertion granularity. Escalate to user with a proposed correction (APPROVE / REJECT). When approved, update `etch-map.yaml` when mappings changed.

### Spec-domain findings

When `auto_resolve_spec` is enabled in [pipeline config](../../artifacts/config.md), Settle auto-resolves `domain: spec` findings:
1. Presents a reasonable default assumption
2. Updates Gherkin spec wording when needed
3. Appends the resolution to `ascertainments.md` with an `[AUTO-RESOLVED]` tag and rationale

When `auto_resolve_spec` is false (default), spec-domain findings escalate to the user for manual resolution.

### Provides hash tracking

After fixes, check whether any target subspec's `Provides:` line changed compared to `provides_hash`. When changed → set `provides_changed: true`. This determines blast radius execution in the next loop.

### Loop limit

Settle loops up to `settle_loops` times (see [pipeline config](../../artifacts/config.md); default: 0; tracked by `loop_count`). When the limit is reached with remaining findings, escalate to user. At default (0), every failure escalates immediately.

## Sub-stage Ordering

```
Settle.Triage → Settle.Scope → Settle.Fix → Settle.Confirm
```

### Settle.Triage

1. Locate most recent `verify_{timestamp}.md`
2. Parse Critical, High, and Medium findings only
3. Identify owning subspec per finding via BID → subspec mapping
4. Group findings by owning subspec

### Settle.Scope

1. Regenerate `delivery-order.yaml` from current `Requires:`/`Provides:` declarations
2. Identify `target_subspecs` — subspecs owning failing BIDs
3. Compute `blast_radius` — transitive downstream dependents of target subspecs
4. Write `rerun_scope` to `pipeline-state.yaml`
5. `_integration` is always included in re-run scope

### Settle.Fix

1. Apply fixes by `resolution_domain` (impl, test, spec) using the policies above
2. After all fixes, check `provides_hash` changes

### Settle.Confirm

1. Re-run Inspect to confirm resolution
2. When failures remain and a loop is needed:
   - `provides_changed: false` → only `target_subspecs` re-run
   - `provides_changed: true` → `target_subspecs` + all `blast_radius` re-run
   - Reset affected subspecs to `pending`; unaffected retain `passed`
3. Route by domain: `impl` → Realize, `test` → Etch, `spec` → Ascertain, mixed → earliest stage

## Exit Checks

### Resolution completeness

- [ ] Every Critical, High, and Medium finding from the verify report is addressed
- [ ] Each fix matches the finding's `resolution_domain` (impl fixes touch `src/`, test fixes follow the three-tier policy, spec fixes append to `ascertainments.md`)

### Test-domain fix compliance

- [ ] Tier 1 fixes preserve test function names and count; test results are identical before and after
- [ ] Tier 2 fixes use only the BID's Gherkin step + test Arrange data for derivation; user was notified
- [ ] Tier 3 fixes were user-approved before application
- [ ] Every test-domain fix traces to a tier in the three-tier policy

### Scope integrity

- [ ] `delivery-order.yaml` was regenerated from current declarations
- [ ] `target_subspecs` matches subspecs owning failing BIDs
- [ ] `blast_radius` includes all transitive downstream dependents
- [ ] `_integration` is in re-run scope
- [ ] `provides_changed` reflects actual hash comparison

### Loop tracking

- [ ] `loop_count` incremented in `pipeline-state.yaml`
- [ ] When `loop_count` reaches `settle_loops` config value: escalated to user, loop halted

### Completion

- [ ] When zero Critical/High/Medium findings remain after re-Inspect: status is **COMPLETE**
- [ ] When findings remain: routed to correct stage by domain, with correct subspec scoping

### Boundary discipline

- [ ] `impl` fixes are scoped to production code only
- [ ] `test` fixes follow the three-tier policy — each fix traces to a specific tier
- [ ] When `auto_resolve_spec` enabled: `spec` fixes append `[AUTO-RESOLVED]` to `ascertainments.md` with rationale
- [ ] When `auto_resolve_spec` disabled: `spec` findings escalated to user
- [ ] Settle creates new BIDs, subspecs, and test functions only through stage re-entry (Etch/Inscribe), and only within scope
- [ ] Finding severity classifications match the verify report — Settle processes each finding at its reported severity
