# Delivery Order

The sequenced list of subspecs for implementation. Produced by Layout, consumed by the pipeline orchestration layer to sequence Etch and Realize execution.

## What It Contains

A YAML file listing subspecs in implementation order with dependency edges:

```yaml
feature_id: "{feature_id}"
generated_at: "2026-03-10T14:30:00Z"
subspecs:
  - file: "users.feature"
    provides: ["UserRecord"]
    requires: []
    depends_on: []
  - file: "auth.feature"
    provides: ["AuthToken"]
    requires: ["users.feature -> UserRecord"]
    depends_on: ["users.feature"]
  - file: "_integration"
    provides: []
    requires: ["*"]
    depends_on: ["users.feature", "auth.feature"]
```

Subspecs are identified by filename. Each entry declares its dependencies on other subspecs. The `_integration` pseudo-subspec entry uses a wildcard dependency to depend on all other subspecs.

## Lifecycle

Compiled (derived) from subspec `Requires:` and `Provides:` declarations. First generated by Layout after subspecs are created and validated. Regenerated by Settle.Scope before blast-radius calculation when a Settle loop is initiated. `depends_on` edges are mechanically computed from `Requires:` references. Kept after the feature completes as an inspection record.

## Derivation Rules

- Parse each subspec's `Requires:` line; the file reference before `->` becomes a `depends_on` entry
- A subspec with no `Requires:` is a root node (empty `depends_on`)
- `_integration` depends on all other subspecs (wildcard)
- Topological sort on `depends_on` determines implementation order
- A cycle in `depends_on` is an error surfaced at Layout.Verify (ANLZ-005)
- `generated_at` records when the derivation ran
